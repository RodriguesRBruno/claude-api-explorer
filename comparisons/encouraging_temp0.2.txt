**Question 2 of 9:**

Which of the following is a valid way to write a multi-line comment in Python?

**A)** `/* This is a comment */`

**B)** `# This is a comment`

**C)** `// This is a comment`

**D)** A triple-quoted string: `"""This is a comment"""`

What's your answer?


💡 Hint: Sure! Here's a helpful hint:

Think about what you know about Python comments:
- Single-line comments use `#`
- Python doesn't have a dedicated multi-line comment syntax like some other languages (C, Java, JavaScript)

Now, look at the options:
- Options A and C use comment styles from **other programming languages** (C-style and JavaScript-style). Do those work in Python?
- Option B is a single-line comment, not multi-line.
- Option D uses triple quotes (`"""` or `'''`). What are triple-quoted strings typically used for in Python? Think about docstrings...

That should point you in the right direction! 😊


Not quite! Let me explain:

**The correct answer is D) A triple-quoted string: `"""This is a comment"""`**

Here's why: Option B (`# This is a comment`) is actually a **single-line** comment, not a multi-line one. While you *can* use multiple `#` symbols on separate lines for multi-line comments, the question asks for a single valid way to write a multi-line comment.

Python doesn't have a dedicated multi-line comment syntax like `/* */` (C-style) or `// ` (JavaScript-style). Instead, the convention is to use **triple-quoted strings** (`"""..."""` or `'''...'''`). These are technically string literals, but when they're not assigned to a variable or used as function arguments, they act as comments and are ignored by Python.

**Key takeaway:** Triple-quoted strings are the Pythonic way to write multi-line comments and docstrings!

**Question 3 of 9:**

What is the output of the following code?
```python
name = 'Alice'
print('Hello,', name, end='!\n')
```

**A)** Hello, Alice!

**B)** Hello,Alice!

**C)** Hello, Alice!\n

**D)** Hello, Alice !

What's your answer?


💡 Hint: Great! Here's a helpful hint:

Focus on two things in the `print()` function:

1. **The `sep` parameter (default):** By default, `print()` separates multiple arguments with a **space**. So `'Hello,'` and `name` will have a space between them.

2. **The `end` parameter:** This controls what character(s) are printed at the very end. Instead of the default newline, this code uses `end='!\n'`. What do you think that means?

So think about it step by step:
- What gets printed from the arguments?
- What gets printed at the end (after all arguments)?

That should help you figure it out! 😊


Excellent! That's correct! 🎉

**The correct answer is A) Hello, Alice!**

Here's the breakdown:
- `print('Hello,', name, end='!\n')` has two arguments: `'Hello,'` and `name` (which is `'Alice'`)
- By default, `print()` separates arguments with a **space**, so it prints: `Hello, Alice`
- The `end='!\n'` parameter replaces the default newline with `!` followed by a newline (`\n`)
- So the final output is: `Hello, Alice!` (with a newline at the end)

**Key takeaway:** The `end` parameter in `print()` lets you customize what character(s) appear at the end of the output. The default is `'\n'` (newline), but you can change it to anything you want!

**Question 4 of 9:**

Given the code below, what does `z` hold after execution?
```python
x = 5
y = x
x = 10
z = y
```

**A)** 5

**B)** 10

**C)** 15

**D)** Raises a NameError

What's your answer?


💡 Hint: Great! Here's a helpful hint:

Think about how variable assignment works in Python. When you write `y = x`, you're not creating a "link" between `y` and `x`. Instead, `y` gets a **copy of the value** that `x` currently holds.

Now trace through the code step by step:
1. `x = 5` — x now holds 5
2. `y = x` — y gets the value 5 (at this moment)
3. `x = 10` — x is reassigned to 10. Does this affect y?
4. `z = y` — z gets whatever value y is holding

The key question: When you change `x` to 10, does that change what `y` is holding? 🤔

That should help you work through it!


Not quite! Let me explain:

**The correct answer is A) 5**

Here's the step-by-step breakdown:
1. `x = 5` — x holds 5
2. `y = x` — y gets the **value** 5 (a copy, not a reference to x)
3. `x = 10` — x is reassigned to 10, but this does **not** affect y. y still holds 5!
4. `z = y` — z gets the value that y is holding, which is 5

**Key takeaway:** In Python, integers are **immutable** and assignment creates a copy of the value, not a link between variables. When you reassign `x`, it doesn't change what `y` holds. They're independent after the assignment.

This is different from how references work in some other languages, but in Python with simple values like integers, each variable holds its own copy.
